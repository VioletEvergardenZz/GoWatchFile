# Roadmap（2026）

> 文档状态：现行（主线）  
> 校准日期：2026-02-24
> 口径说明：该文档为主线口径，发生边界变更时需优先更新。

## 1. 规划原则

- 以“当前真实能力”推进，不以历史叙事驱动优先级。
- 主线能力优先于支撑能力。
- 每个阶段必须有明确可验收产出（接口、指标、运行手册）。

## 2. 阶段路线

### 阶段 A（当前到下个 1~2 个迭代）

目标：稳定运维事件闭环主线，完成收口。

- 文件入云：提升采集稳定性、回压保护、失败重试与指标可观测。
- 告警决策：提升规则治理稳定性与抑制/升级解释性。
- AI 分析：固化成功/重试/降级/失败分类门禁，保证可回放。
- 知识库复用：稳定引用式问答与推荐链路，完善评估门禁。
- 控制台：确保发现、诊断、回放、追溯路径统一可用。

### 阶段 B（随后 2~4 个迭代）

目标：在不扩边界前提下增强支撑能力，提升可运维性。

- 控制面：提升 Agent 分组治理与任务分发稳定性（支撑主线）。
- 系统资源：完善主机视图、权限边界与告警联动样例。
- 指标：统一 SLO 口径和阶段门禁，补齐主线可观测面板。
- 文档：保持主线与专题目录口径一致，避免回归漂移。

### 阶段 C（中期）

目标：扩展平台能力，但不影响已收口主线。

- 多输入源适配策略（文件之外的事件输入）。
- 更完整的策略编排与任务路由（先小规模灰度）。
- 安全治理升级（权限分层、审计深度、密钥治理）。
- 跨环境发布与容量治理体系。
- 复杂多用户、多租户、细粒度 RBAC/计费进入评估与设计阶段。

## 3. 主线能力与支撑能力的迭代预算

- 主线能力（文件入云/告警决策/AI 分析/知识库复用/统一控制台）：约 80%
- 支撑能力（控制面/系统资源/通知通道）：约 20%

## 4. 里程碑验收标准

每个里程碑至少满足：

1. 关键接口可回归。
2. 关键指标可观测。
3. 操作手册可执行。
4. 回滚路径明确。

## 5. 非目标声明（当前阶段）

- 不以“复杂多用户能力上线数量”作为当前阶段成功指标。
- 不以“页面数量增长”作为平台化完成标准。
- 不引入脱离当前运维场景的复杂编排系统。
