# 文档地图

> 文档状态：治理基线
> 更新时间：2026-02-24

## 1. 目标

当前项目已明确为“面向 SRE/运维的运维事件闭环平台”。文档体系必须支持以下目标：

- 让新接手的运维工程师在 30 分钟内理解平台主线能力。
- 让值班 SRE 能快速定位到对应模块的运行手册与排障入口。
- 让平台维护者能基于同一份能力边界做迭代，不再出现多版本口径冲突。

## 2. 文档结构（现行）

1. `docs/01-平台定位/`
定义平台是什么、服务谁、主线能力与支撑能力是什么。

2. `docs/02-架构设计/`
定义系统如何运行、模块如何协作、当前约束在哪里。

3. `docs/03-能力模块/`
定义每个能力模块的职责边界、输入输出、主要接口与不做范围。

4. `docs/04-路线图与计划/`
定义演进节奏、阶段目标、当前迭代 TODO 与优先级。

5. 专题目录（兼容保留）
`docs/02-开发运维/`、`docs/03-告警与AI/`、`docs/04-知识库/`、`docs/05-指标与评估/` 继续承载运行手册、样本与评估报告。

## 3. 目录序号说明

1. `docs/` 当前采用主线与兼容并存的双轨策略，因此会出现 `02/03/04` 同序号目录。
2. 主线目录定义平台边界，专题目录只承载操作细节与评估资产。
3. 是否“最新”不由序号判断，统一以 `docs/00-文档治理/文档现行状态清单.md` 为准。

## 4. 读者视角导航

- 运维工程师：
  - 先看 `docs/01-平台定位/项目定位与主线能力.md`
  - 再看 `docs/03-能力模块/功能模块说明.md`
  - 最后按模块进入专题运行手册。

- SRE：
  - 先看 `docs/02-架构设计/总体架构说明.md`
  - 再看 `docs/05-指标与评估/监控指标字典-v1.md`
  - 最后看 `docs/04-路线图与计划/TODO.md` 确认近期风险与待办。

- 平台维护者：
  - 先看 `docs/02-架构设计/总体架构说明.md`
  - 再看 `docs/03-能力模块/功能模块说明.md`
  - 然后看 `docs/04-路线图与计划/Roadmap.md` 与 `TODO.md`。

## 5. 更新触发规则

出现以下变化时，必须更新主线文档：

- 新增或下线一级能力模块。
- 主线能力与支撑能力发生调整。
- 架构中新增关键数据存储或控制链路。
- Roadmap 优先级发生跨阶段调整（例如从 P1 提升到 P0）。
- API 契约、降级语义、指标口径发生变更。

## 6. 单一事实源

- 能力定位：`docs/01-平台定位/项目定位与主线能力.md`
- 架构边界：`docs/02-架构设计/总体架构说明.md`
- 模块职责：`docs/03-能力模块/功能模块说明.md`
- 迭代优先级：`docs/04-路线图与计划/TODO.md`
- 接口变更同步检查：`docs/00-文档治理/接口变更文档同步检查清单.md`
