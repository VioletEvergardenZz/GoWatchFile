# 项目定位与主线能力

> 文档状态：现行（主线）  
> 校准日期：2026-02-22
> 口径说明：该文档为主线口径，发生边界变更时需优先更新。

## 1. 平台定义

GWF 是一个面向运维工程师 / SRE / 平台维护者的运维平台，目标是把分散的运维事件处理流程收敛为可观测、可追溯、可执行的统一闭环。

平台不再以“文件监控上传”作为唯一主线，而是以“控制、监控、决策、知识复用”作为长期主线。

## 2. 目标用户

- 运维工程师：需要快速执行日常巡检、处置告警、回放问题。
- SRE：需要稳定性指标、告警治理、故障复盘链路。
- 平台维护者：需要跨模块迭代、版本治理、能力边界控制。

## 3. 核心能力与辅助能力

### 3.1 核心能力（平台主线）

1. Agent 运行与管理（控制面）
2. 告警监控与决策
3. 系统/主机资源采集
4. 运维知识库
5. 平台化控制台能力

### 3.2 辅助能力（非主线但保留）

1. 文件监控与上传（输入适配器）
2. 通知通道（钉钉、邮件）
3. AI 摘要与推荐（增强能力，默认可降级）

## 4. 当前版本能力口径（以代码为准）

- 后端 `go-watch-file` 提供统一 API，内含控制面、告警、系统资源、知识库、文件采集能力。
- 控制面已具备 Agent 注册/心跳/摘流、Task 创建/拉取/执行回传、审计日志、SQLite 持久化。
- 知识库已具备文章生命周期、检索、问答、推荐、文档导入、回滚与评审相关接口。
- 告警模块已具备规则匹配、抑制、升级与控制台配置能力。
- 系统资源采集支持主机概览、资源指标、磁盘、进程视图与进程终止接口。
- 文件监控上传模块作为输入侧能力继续维护，不再作为架构主线。

## 5. 当前边界（明确不做）

- 当前版本不是完整的分布式调度平台，控制面仍以单实例后端 + SQLite 为主。
- 当前版本未引入多租户隔离与 RBAC 体系。
- 文件适配器当前仅覆盖本地文件监控场景，不覆盖全量输入源生态。

## 6. 判断“是否属于主线需求”的标准

一个需求满足以下任一条件，归为主线需求：

- 直接提升告警决策闭环稳定性或可操作性。
- 直接提升 Agent/任务控制闭环可观测性与可靠性。
- 直接提升知识复用效率与故障处置效率。
- 直接提升平台维护者对系统运行状态的全局控制能力。

否则归入辅助能力需求，按主线优先级让位。
