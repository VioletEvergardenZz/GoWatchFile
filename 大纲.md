# GWF 平台蓝图（现行）

> 文档状态：历史背景（兼容保留）  
> 校准日期：2026-02-22
> 口径说明：历史材料用于追溯，不作为当前主线能力定义依据。

## 1. 蓝图目标

将 GWF 持续演进为可落地的运维平台，而不是单一文件上传工具。  
平台蓝图围绕以下三类结果组织：

1. 可执行：可以在控制台发起、跟踪、回收运维任务。
2. 可决策：可以基于告警与知识做处置建议和闭环记录。
3. 可维护：可以持续观测系统状态并完成迭代治理。

## 2. 主线能力蓝图

1. 控制面与 Agent 运行管理
2. 告警监控与决策治理
3. 系统/主机资源采集与轻量处置
4. 知识库沉淀、检索与引用式问答
5. 平台控制台统一视图与操作入口

## 3. 辅助能力蓝图

1. 文件监控上传作为输入适配器长期维护。
2. 通知通道作为告警执行补充能力。
3. AI 作为增强层，默认支持降级兜底。

## 4. 架构蓝图

- 前端 `console-frontend` 提供多视图控制台。
- 后端 `go-watch-file` 提供统一 API 与执行逻辑。
- 本地 SQLite 支撑控制面与知识库持久化。
- 本地状态与配置文件支撑运行态与兼容恢复。
- OSS 负责对象存储目标。

## 5. 迭代蓝图

- 短期：收敛核心闭环稳定性，补齐文档与运维门禁。
- 中期：增强控制面治理能力与跨模块联动能力。
- 长期：扩展输入源、权限治理、跨环境发布与容量体系。
